# 文档规范

状态：Active

本文定义 `configer` 项目的文档写作和维护规范。后续项目说明书、产品设计、技术方案、模块文档都应遵守本文。

## 文档分类

| 分类 | 目录 | 说明 |
|---|---|---|
| 项目级 | `docs/00_project/` | 背景、目标、范围、术语、路线图 |
| 产品级 | `docs/10_product/` | 用户场景、页面流程、功能需求、验收 |
| 技术级 | `docs/20_architecture/` | 技术方案、模块架构、数据生命周期 |
| 研发级 | `docs/30_development/` | 文档、代码、测试、交付规范 |
| 模块级 | `docs/40_modules/` | equip、item、language 等模块入口和摘要 |
| 参考级 | `docs/90_reference/` | Excel 协议、源表清单、外部流程 |

## 命名规则

- 目录名使用两位数字前缀加英文名称，例如 `20_architecture/`。
- 文档文件名使用英文小写和下划线，例如 `project_directory_design.md`。
- 文档标题可以使用中文，标题必须准确描述主题。
- 一个文档只承担一个职责；不要把产品设计、技术方案、实施计划混写在同一文件。

## 文档头部

每份规范性文档顶部必须包含状态：

```markdown
# 文档标题

状态：Draft | Active | Superseded
```

- `Draft`：草案，不能作为开发验收依据。
- `Active`：当前有效，开发和评审应遵守。
- `Superseded`：已被新文档替代，必须写明替代文档链接。

## 写作规则

- 先写结论，再写依据。
- 每个规则必须可执行，避免“优化体验”“适当处理”这类不可验收表述。
- 涉及 Excel 的文档必须说明表路径、sheet、字段来源、是否允许写回。
- 涉及 UI 的文档必须说明入口、用户操作、状态反馈和失败提示。
- 涉及架构的文档必须说明依赖方向，避免模块互相调用。
- 引用已有资料时使用相对链接，不复制大段内容。

## 变更规则

- 修改 `Active` 文档时，应同步检查依赖它的模块入口。
- 如果新决策推翻旧资料，应在旧文档顶部标记 `Superseded` 或在模块入口记录差异。
- 不移动 `docs/equip/doc/` 现有文件，除非另有明确迁移计划。
- 开发前发现文档冲突，应先澄清权威来源，再动代码。

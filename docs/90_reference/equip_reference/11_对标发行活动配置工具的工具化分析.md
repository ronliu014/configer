# 11 · 对标发行活动配置工具：装备配表的工具化分析

> 把"装备配表"放到**发行活动配置工具（DDGPM 发行管理系统）**的设计范式下重新解构：找出二者的同构关系、做概念映射、给 equip 每个字段标注"对标控件类型"，并推演一个"装备配置工具"该长什么样。
> 本篇是分析/对标文档，**不修改** 01~10 任何已有文档；字段明细取自 [05 字段清单](05_字段清单表.md) 与 [09 关联表字段清单](09_关联表字段清单.md)，配置流程见 [10 装备配置流程](10_装备配置流程.md)。

---

## 一、为什么能对标——两者是"同构"的

发行活动配置工具的设计内核，一句话：

> **策划只手填少量"维度"，系统通过"逻辑生成控件"自动拼出所有关联 ID 与派生字段，再把数据分发到多张关联表。**

这恰恰就是 equip 主表现在用 **Excel 公式 + Sheet4 字典**在做的事。区别只在"谁来做逻辑生成"：

| 维度 | 发行活动配置 | 装备配表（现状） |
|------|-------------|----------------|
| 谁做"逻辑生成" | 平台工具，配置时实时算 | Excel 公式，引用 Sheet4 |
| 策划面对的东西 | 表单控件 | 一行带几十列公式的 Excel |
| 关联表填充 | 工具按规则写各模板表 | 公式拼 ID，人去关联库核对 |
| 防错机制 | 提交前配置比对 + 审核 | 人工核对"悬空外键"（易漏） |

**结论：装备配表本质上是"还没工具化的发行活动配置"。** 用它的范式套，几乎每个概念都能一一对上。

## 二、概念映射表（发行活动配置 → 装备）

| 发行活动配置的设计概念 | 装备配表里的对应物 |
|---|---|
| **模版（Template）** | equip 主表列结构 + Sheet4 字典（定义"装备这个对象长什么样"） |
| **活动类型**（多人跑酷 / 战令） | **装备部位 × 转数 × 品质** 的维度组合（不同组合 = 不同"配置形态"） |
| **手填控件**（文本框/下拉/道具） | equip 的 **9 个手填维度**（部位/职业/转数/分支/品质/第几套/挂件/图标 等） |
| **逻辑生成控件（999）** | equip 的**公式列**（装备ID/各库ID/职业限制/穿戴等级/名称Key…） |
| **组合控件**（时间类型、服务器区间随类型变形） | equip 的**品质→随机词条条数、转数→等级**这类联动 |
| **模板表**（multimatch / shop / activity_goal） | equip 的**关联库**（proplib / job_group / random_proplib / special_drop / suit） |
| **语言包控件** | language 表（名称Key / 描述Key / 套装名Key） |
| **道具控件** | item 表（非绑 + 绑定两条道具） |
| **配置比对 / 校验** | **悬空外键校验**（自动拼出的 ID 必须在对应库里存在） |
| **审核 / 发测试 / 发正式 / 状态流转** | 装备当前**缺失**的一环（详见第六、七节） |

## 三、控件类型词表（equip 版，对齐发行活动配置 10 种控件 + 扩展）

发行活动配置最有价值的抽象，是把每个字段归到一种**控件类型**。把这套词表搬到装备上：

| 代号 | 控件类型 | 沿用/扩展 | 在装备里指什么 |
|------|---------|----------|----------------|
| **2** | 固定值 | 沿用 | 全表恒定、不让策划改（战力类型=1、战力计算=6） |
| **3-int** | 文本框·整型 | 沿用 | 整数维度（几级的装备、掉落等级<遗留>） |
| **3-res** | 资源选择器 | 扩展（文本框+资源库） | 挂件ID、装备图标（参照 Sheet4 资源清单选一个） |
| **4-1** | 下拉单选 | 沿用 | 部位/基础职业/转数/分支/品质（**选项即 Sheet4 字典**） |
| **5** | 道具控件 | 沿用 | item 两条道具的批量生成（间接） |
| **6** | 语言包 | 沿用 | 名称语言包、描述语言包 |
| **7** | **关联库引用** | **扩展⭐** | 选一条**已存在**的库条目（套装库 SuitId、技能随机库 RandonSkillID）。装备"重关联"，比活动多这一类 |
| **8** | **特效/套装开关** | **扩展⭐** | 现在藏在「备注」文本里的"特效""套装"标记位，应显式化为开关；开关驱动 DropID / 图鉴套装ID 的逻辑生成 |
| **999** | 逻辑生成（含组合） | 沿用 | 所有公式列 + 自洽编码外键（系统拼接，策划不填） |
| **0** | 空 | 沿用 | 遗留无用字段（投放展示页签） |

> ⭐ **7 关联库引用** 和 **8 特效/套装开关** 是装备比活动配置**多出来**的两类控件——这正是装备"重关联、靠备注关键字开功能"的特点，是工具化时要重点设计的地方。

> **命名口径**：本表数字代号（2 / 3-int / 4-1 / 7 / 999…）仅作"对标发行活动配置工具"的**对照索引**保留；工具实现与 schema 正文**统一用 [13 §3.6](13_字段配置标准（表与字段Schema）.md) 的语义名**（select / int-input / resource / lang / ref-picker / switch / gen / fixed / hidden）。两套映射见 13 §3.6 表。

### 4.1 按控件类型归总（一眼看清"工具该放几个输入控件"）

| 对标控件类型 | 命中字段 | 数量 | 工具化含义 |
|---|---|---|---|
| **4-1 下拉单选** | 部位、基础职业、转数、1-2转分支、3转分支、品质 | 6 | 选项全来自 Sheet4，**表单核心输入区** |
| **3-int 文本框·整型** | 几级的装备、掉落等级(遗留) | 2 | 数字维度 |
| **3-res 资源选择器** | 挂件ID、装备图标 | 2 | 接资源库，可预览 |
| **6 语言包** | 名称语言包、描述语言包 | 2 | **可直接复用活动配置已有控件** |
| **7 关联库引用** | 初始套装库、装备技能随机库 | 2 | 选已存在库条目（装备特有）；装备技能随机库为二级链 →equip_skill_drop→equip_skill_effect |
| **8 特效/套装开关** | （藏在备注 Remark 里）特效、套装 | 2 | **应从文本标记位拆成显式开关** |
| **2 固定值** | 战力类型、战力计算 | 2 | 锁定不可改 |
| **equip_rare 映射类** | 道具品质、强度子品质 | 2 | 按品质映射 equip_rare 取值，**可下钻查看映射行（非真外键）** |
| **999 逻辑生成** | 其余约 33 列（公式/自洽编码外键） | ~33 | **系统拼接，策划完全不碰** |
| **0 空** | 投放展示页签(遗留) | 1 | 隐藏 |

**一句话**：策划真正要操作的输入控件 ≈ **6 个下拉 + 2 个数字/资源 + 2 个语言包 + 2 个关联库引用 + 2 个开关**，其余 35 列全部 999 逻辑生成。这与发行活动配置"少手填、多逻辑生成"的形态完全一致。

## 五、关联库字段的控件归类（基于 09）

关联库（equip_proplib / job_group / random_proplib / special_drop / suit / skill_drop）在工具里对应活动配置的**模板表区块**。按"可否在表单里增减行"分两型（正是活动配置 activity_goal vs shop 的区别）：

| 关联库 | 对标活动配置 | 表单形态 | 关键输入控件 |
|---|---|---|---|
| equip_proplib（固定属性） | 类 activity_goal | **不可增减行**，按规则一行 | 属性值"属性ID,数值"（道具/数字组合） |
| equip_random_proplib（随机词条） | 类 shop | **可增减行 + 权重排序** | 属性id(关联prop)、基准值、权重、品质 |
| equip_job_group（职业组） | 类 multimatch | 选职业集合 | 职业ID(多选关联 soul)、基础职业/转数/分支 |
| equip_special_drop + equip_special_prop（特效） | 类 shop（带二级） | 可增减 + 权重 | 属性ID(关联special_prop)、权重；效果=技能ID(关联skill) |
| equip_suit（套装） | 类 activity_goal | 一条 | 套装名(语言包)、套装总数 |
| equip_skill_drop + equip_skill_effect（装备/套装技能） | 类 shop（带二级） | 可增减 + 权重 | 候选技能(按技能组ID分组)、权重；效果=关联技能效果编号(EquipSkillEffectID)→equip_skill_effect（技能ID关联skill、套装触发件数、属性值、评分等） |

> 二级依赖（job_group→soul、special_drop→special_prop→skill、random_proplib→random_proplib_group、skill_drop→skill_effect→skill）= 活动配置里"组合控件的下钻数据源"。逐字段见 [09](09_关联表字段清单.md)。

## 六、把装备流程套进发行活动配置的功能模块

发行活动配置分 **模版管理 / 活动管理（表单+发布）/ 审核管理** 三大块。装备配置流程（[10](10_装备配置流程.md)）可同构组织为：

**① 模版管理 → 装备模板定义**
- 定义 equip 列结构 + 每列控件类型（见第四节）；Sheet4 字典 = 模板的"下拉数据源 + 逻辑生成规则库"。
- 不同"部位×转数×品质"是模板的**变体**（类比活动类型），但结构不变（详见第七节差异）。

**② 活动配置表单 → 装备配置表单**
- 策划只填第四节标 **4-1/3/6/7/8** 的那十来个控件（对应"基础信息区块"）。
- 系统实时逻辑生成 35 列：装备ID/各库ID/等级/名称Key（对应"逻辑生成控件"）。
- 关联库区块（proplib/random_proplib/special_drop/suit/skill_drop）= 活动配置的 activity_goal/shop 区块，按第五节"可否增减行"渲染。

**③ 配套表 → item + language**
- item 两条道具（非绑+绑定）= 道具控件批量生成，ID 差"8"前缀、L/M 互指可由工具自动完成。
- language 文案 = 语言包控件，**逻辑可直接复用活动配置已有实现**。

**④ 校验 → 配置比对（收益最大的复用点）**
- 活动配置在审核环节做**全量表配置比对**；装备的**悬空外键校验**（[10](10_装备配置流程.md) Step 2）本质同一件事——提交前自动扫一遍：每个自动拼出的 ID 在目标库里是否存在。
- 现状靠人逐张核对，工具可"一键校验 + 高亮缺失库条目"。

**⑤ 审核 / 发布（装备目前缺的一环）**
- 活动配置有完整的"发测试→发正式→审核→状态流转"；装备现在直接改 Excel 提交 trunk（参考 `excel-batch-commit` 流程），**无审核态**。
- 涉及悬空外键风险时，"提交前比对 + 审核"价值很高，可向活动配置借这套设计。

## 七、关键差异（装备特有，不能照搬）

1. **重关联、多级依赖**：活动配置基本是 1 主表 + 5 平级模板表；装备是 **17 张表、有二级依赖**（job_group→soul、special_drop→special_prop→skill、skill_drop→skill_effect→skill）。工具化时**关联库"按需补条目"流程**远比活动复杂——这是装备的核心难点。
2. **自洽编码 vs 流水号**：活动配置逻辑生成多是"地区+模板号+活动号"流水拼接；装备 ID 是**维度自洽编码**（拿维度一拼正好命中对方已存在的 ID）。既是优点（少手填），也是**悬空外键的风险源**（漏建库就指空）。
3. **维度驱动变形，而非多模板**：活动配置切活动类型 = 换一套模板表；装备换部位/品质只是换维度值、模板结构不变。所以装备更适合 **"单模板 + 维度驱动"**，而不是活动那种"多模板并存"。
4. **靠备注关键字开功能**：特效/套装当前是写在「备注」文本里的关键字（脆弱、易错），工具化必须把它拆成第三节的 **8 特效/套装开关**。

## 八、工具化落地建议与路线

按投入产出比排序：

1. **【已随本篇完成】给字段标控件类型** — 第四节已把 equip 55 列对齐到控件词表；下一步可同样给 [09](09_关联表字段清单.md) 的关联库字段补一列控件类型，形成完整"模板定义"。
2. **【最高优先 · 最独立】悬空外键校验工具** — 输入 equip 一行的维度，按 [04 ID编码规则](04_ID编码规则速查.md) 反推应存在的各库 ID，逐个核对关联库，高亮缺失。不依赖整套工具即可单独上线，直接替代人工核对。
3. **关联库"按需补条目"决策树** — 把"什么维度组合 → 要补哪些库"画成规则引擎（第五节 + [10](10_装备配置流程.md) 第三节"前置库"），作为工具核心。
4. **配置表单 MVP** — 先做 equip 本体的"维度输入 + 实时逻辑生成 + 校验"，item/language 配套自动生成，套装/特效作为开关。
5. **（可选）审核 / 发布态** — 借活动配置的状态流转，给装备提交加"比对 + 审核"。

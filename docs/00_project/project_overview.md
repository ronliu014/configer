# 项目说明书

状态：Active

## 背景

DD 游戏项目的策划配置大量依赖 Excel 表。装备、物品、文案、关联库之间存在复杂字段公式、ID 编码规则和跨表引用关系。当前手工配置成本高，容易出现漏配、错配、悬空引用、无效数据和冗余数据。

`configer` 的目标是把这些配表规则沉淀为可视化、可校验、可安全写回的本地配置工具，让策划在工具中完成配置任务，并让系统自动检查配置链路。

## 目标

- 自动化完成重复、复杂、公式驱动的 Excel 配置工作。
- 快速发现配置链路中的缺失、错误、冲突、无效数据和冗余数据。
- 用统一框架承载多条业务配置线，当前先落地 `equip`，后续扩展到 `item`、`role`、掉落、技能等模块。
- 保留现有 Excel 配表工作流，特别是公式列、格式、批注和多 sheet 内容。

## 用户与使用场景

主要用户是游戏策划和配置维护者。典型场景包括：

- 新增一件装备，并同步维护配套物品和文案。
- 修改装备维度后，检查关联库、物品、语言包是否仍然有效。
- 批量查看装备配置中存在的悬空引用或异常数据。
- 写回本地配表前预览变更，并生成 changelog 作为提交说明。

## 范围

当前 MVP 以装备系统为首个业务线，范围包括：

- 可增删改：`equip`、`item`、`language`。
- 只读引用与校验：装备相关一级关联表，例如职业组、属性库、随机词条库、特效库、套装库、稀有度、装备技能掉落等。
- 加载和写回：本地 Excel 配表，优先支持用户选择配置目录。
- 校验：MVP 只做引用主键存在性校验；完整链路对账放到后续版本。

## 非目标

MVP 不做以下内容：

- 不做所有业务配置线的完整平台化接入。
- 不编辑或补建装备关联表。
- 不做完整对账三态、反向索引和写时算性能架构。
- 不做技能子系统、掉落、活动、天赋树等外围系统。
- 不做自动 SVN 提交或后台服务。
- 不把工具 demo 的 mock 逻辑当作真实业务逻辑。

## 阶段路线

- v1.0：装备 MVP，完成三表增删改、关联只读引用校验、安全写回和 changelog。
- v1.1：完善对账模型，支持关联表补建和配置状态三态。
- v1.2：补齐图标资源、批量导入、大表性能和更完整的装备链路编辑体验。
- v2.0：接入 AI 辅助规则编写、自动提交流程和多业务线平台化能力。

## 成功标准

v1.0 成功标准：

- 能选择本地配置目录并加载真实 `equip`、`item`、`language` 和装备关联表。
- 能新增、编辑、删除一件装备及其配套物品、文案。
- 能显示关联表条目，引用不存在时给出明确异常。
- 写回后 Excel 公式列仍是公式，格式、批注和非目标 sheet 不丢失。
- 每次写回前有备份，每次写回后有可读 changelog。
- 修改 `equip` 不影响 `item`、`language` 或未来业务模块的独立演进。

## 风险与约束

- Excel 写回是最高风险点，必须只写手填字段，不得整表重建。
- 配置规则必须以 `docs/40_game_config/` 中的配置说明为依据，不得从 demo 反推逻辑。
- 浏览器本地文件访问依赖 File System Access API，MVP 以 Chromium 系浏览器为目标。
- `source/` 是本地资源目录认知，不应成为工具运行的固定依赖；真实使用时应由用户选择配置目录。

## 五、页面规范

### 5.1 装备列表页（列表页 + Tab 分类）

```
[全部][战士][弓箭手][魔法师][牧师]                    ← 职业分类 Tab
─────────────────────────────────────────────────
搜索[装备ID/备注____] 品质[全部▾] 转数[全部▾] 部位[全部▾] 配置状态[全部▾] [查询][重置]
[+ 添加装备] [⬆ 批量添加]                            ← 操作行（筛选栏下方）
─────────────────────────────────────────────────
图标 | 装备ID | 备注 | 部位 | 职业 | 转数 | 品质 | 穿戴等级 | 标记 | 操作
🟩  |30110…|战士0转1级绿武器|武器|战士|0|●绿|1|●已配齐|详情 编辑 ⋯
…  分页器：共 N 条  ‹ 1 2 3 … ›
```

- **分类 Tab**：基础职业 全部 / 战士 / 弓箭手 / 魔法师 / 牧师。
- **搜索**：单框，模糊匹配 `装备ID` + `备注(Remark)`。
- **筛选**：品质 / 转数 / 部位 三个下拉（可扩展 分支、第几套）+ **配置状态**（全部 / 已配齐 / 待补 / 数据异常，见第十一节）；`[查询][主] [重置][次]`。
- **操作行**（筛选栏下方、表格上方）：`[+ 添加装备][主]` + `[⬆ 批量添加][次]`。
  - **添加装备**：进入单件两阶段添加流程（见 5.2）。
  - **批量添加**：弹窗上传一个 Excel（每行一件装备的基础维度数据），导入后系统按编码规则批量生成装备行并逐条进入「配置补全」对账。**当前 demo 仅展示上传弹窗样式**（拖拽区 + 导入说明 + 模板下载占位），不接实际解析/导入逻辑（逻辑以文档为准，见 §十二）。
- **列**：图标（见 5.4）/ 装备ID（可点→详情）/ 备注 / 部位 / 职业 / 转数 / 品质（色标签）/ 穿戴等级 / **标记（只示配置状态三态：已配齐 / 待补 N 项 / 数据异常，见 §七）** / 操作（详情·编辑·折叠 ⋯ 内放复制·删除）。
- **数据量大**：分页，每页 20 条；删除二次确认。
- **性能**：列表「配置状态」**只读预存字段**，不在列表加载时实时跑链路对账（见第十一节）。

### 5.2 添加装备页（两阶段 · 自上而下 ★）

> **关键修正**：实际配置是**自上而下**——策划**先填维度建出 equip 行**，再回头补关联库 / item / language（[10](10_装备配置流程.md) 的"先备库"是依赖顺序，不是操作顺序）。所以添加页拆成两步：先落库，再补全。

**第 1 步 · 填维度（双栏）**

```
┌─ 左：维度输入（手填控件）──┬─ 右：系统逻辑生成（只读预览，实时刷新）─┐
│ 基础维度（下拉/整型/资源） │ 装备ID、种类、真实/穿戴等级、属性库ID…   │
│ 特效·套装 开关             │ 名称Key / 随机库串 / 各关联ID …        │
└──────────────────────────┴──────────────────────────────────────┘
底部：[保存装备，进入配置补全 →] [取消]
```
- 左栏只放手填控件（必填红 `*`）；右栏随输入**实时重算**所有逻辑生成字段。
- 点保存 → **equip 行先落库**（即便关联库尚不存在/悬空也能存），进入第 2 步。**新增**时撞 ID（种类相同）拦截提示；**编辑**时复用本表单、带入原维度——保存**放行自身 ID**（没改维度也能存）、**更新原行而非新增**（改了维度则连 ID 一并更新，仍对其他装备查撞号）。

**第 2 步 · 配置补全清单（待办）**

```
装备已创建：战士0转1级太古武器 (3011017001)   配置完整度 5/8
① 关联表（库）  ⚠ equip_proplib 101007001  [去补库]
                ✔ equip_job_group 1010      已配置
                ✔ equip_random_proplib …    已配置
② 配套道具 item ⚠ 非绑+绑定两条             [一键生成两条]
③ 文案 language ⚠ 名称/描述                  [补文案]
        [完成，返回列表]  [查看装备详情]
```
- 系统按这件装备的依赖列出**待办清单**，逐项标 `✔ 已配置` / `⚠ 待补` / `⚠ 数据异常`，给出补全入口——这把原"阻塞式校验"重构为"创建后的补全引导"，贴合自上而下真实流程。
- **数据来源**：待办清单**即读 `cfgStatus.missingList / abnormalList`**（保存装备时写时算好的快照，见 §11.2），进入第 2 步**不重算**；补全某项后按 §11.3 触发该单件对账项的**增量更新**，刷新清单与状态。
- **去补库**：弹出该库关键字段的小表单（库ID 只读、按编码规则预填），保存即写入库 → 该项变 ✔。
- **一键生成两条**：自动生成 item 非绑（ItemId=装备ID）+ 绑定（"8"+装备ID）。
- **补文案**：填名称/描述中文 → 写 language。
- 完整度未满时，该装备在列表「标记」列显示配置状态三态（待补 / 数据异常，见 §七）；补齐后转 **已配齐**。

### 5.3 装备详情页（整页 · 2 Tab ★）

- 整页占满主内容区，内部 **2 个 Tab**：`装备配置(equip)` / `道具配置(item)`。
- 字段按三态渲染（手填 / 自动 / 关联🔗）。
- **道具配置 Tab**：非绑 + 绑定两条并排对照（ID 差"8"前缀，BindItemId/UnBindItemId 互指）。
- **语言包字段直接显示中文**（★ 易读优先）：凡指向 language 的字段（名称/描述语言包、道具名字、套装名、品质名、职业名、特效名…），**列表/详情/抽屉一律显示对应中文**，语言包 Key 作弱化小字附在旁边（中文未配时显示「（未配）」）。点击仍可下钻抽屉看完整文案/Key（对齐 [13 §3.5](13_字段配置标准（表与字段Schema）.md) 的 `format:lang-preview`）。
- 文案（language）不单独成 Tab，挂在对应语言包字段的关联抽屉里。

## 六、关联抽屉规范（多级关联 · 面包屑堆叠 ★）

点击任一关联字段 🔗，从右侧滑出抽屉展示该库条目：

```
┌──── 关联库 ───────────────── [×] ┐
│ equip › special_drop › special_prop  ← 面包屑(可回退)
├──────────────────────────────────┤
│ equip_special_prop (PropID=xxx)  [✔ 存在]
│ ─ 字段只读表 ─
│ 特殊效果  skill0024 🔗  ← 仍可点，继续下钻
│ 特殊效果名称  辉耀
└──────────────────────────────────┘
（从右侧滑出，不遮挡整页；宽 ~480px）
```

- **优先 1 级**：默认只完整渲染 1 级关联条目；2 级及以上字段同样标 🔗，点击才下钻。
- **面包屑 push/pop**：同一抽屉内推进/后退，路径如 `equip › special_drop › special_prop › skill`、`equip › job_group › soul`，避免多抽屉铺屏。
- **状态徽标**：条目顶部 `✔ 存在`（绿）/ `✘ 悬空`（红，附「去补库」）。
- 触发字段示例：使用职业限制→job_group、属性库ID→proplib、随机库→random_proplib、特效库→special_drop（→special_prop→skill）、套装库→suit、技能随机库→equip_skill_drop（→equip_skill_effect→skill）、语言包→language、道具互指→item。

## 九、平台信息架构与多系统接入 ★

### 9.1 平台定位

这不是单一的装备工具，而是一个**配置中心**：用一套统一框架，承载多条业务配置线，并对每条线做"维度输入 → 逻辑生成 → 悬空校验"的配置评估。**装备是首个接入的业务线**，后续其它系统按同一套范式接入。

### 9.2 两层结构

```
配置中心
├─ 业务配置线（vertical，各自独立的主表 + 关联库）
│   ├─ 装备（本期）         equip + equip_*（job_group/proplib/random_proplib/special_drop/suit/rare/skill_drop/skill_effect…）
│   ├─ 〔后续系统 A〕（预留）  自有主表 + 自有关联库
│   └─ 〔后续系统 B〕（预留）  …
│
└─ 公共配置层（shared，被多条业务线关联引用，本期只预留入口不做页面）
    └─ item · func · language · prop · skill …
```

- **业务配置线**：每条线有自己的主表（如 equip）+ 私有关联库（equip_*）。彼此独立，互不影响。
- **公共配置层**：item/func/language/prop/skill 等是**跨业务线共享**的基础表——装备引用它们，后续系统也会引用。本期作为关联抽屉的查看/校验目标存在，**不做独立 CRUD 页面**（详见第十节）。

### 9.3 平台级复用能力（框架提供，业务线只填"数据"）

下列能力由平台框架统一实现，**所有业务线零改造复用**——这正是预留空间的意义：

| 平台能力 | 说明 |
|---------|------|
| 列表页范式 | Tab 分类 + 搜索 + 筛选 + 图标列 + 分页 |
| 字段三态 | 手填 / 自动 / 关联🔗（来源四分类驱动） |
| 控件映射 | 文档 11 控件词表 |
| 逻辑生成预览 | 添加页双栏、按编码规则实时拼接 |
| 悬空外键校验 | 反推应存在的关联 ID → 比对库 → ✔/✘ |

### 9.4 新业务线接入清单（插槽 checklist）

## 十一、配置状态与增量校验（性能架构）★

> 解决"列表要展示每件装备的链路配置情况，又不能因此变慢"的矛盾。对账逻辑沿用 [10 文档](10_装备配置流程.md) 的「命中复用 / 缺失补建」模型。

### 11.1 核心原则：状态展示 ≠ 实时计算

- **要展示**：「哪些装备没配齐整条链路」是对账的产出，也是本工具相对裸 Excel 的核心价值，列表必须能看到。
- **不能实时全量算**：一件装备的链路有 6~10 个对账项（见 11.5），列表 N 行 × M 项 × 查库 = N×M 次查找，每次翻页重算会爆。
- **结论**：**配置状态是"写时算、读时取"的派生字段**——只在写操作时计算并落库，列表只读它（每行 O(1)）。

### 11.2 配置状态字段（冗余存储在 equip 行上）

| 字段 | 类型 | 说明 |
|------|------|------|
| `配置状态 cfgStatus` | enum | `已配齐` / `待补` / `数据异常`（三态；优先级见下） |
| `待补项数 missingCount` | int | 缺失（待补建）的对账项数量，用于列表「待补 N 项」红标 |
| `缺失清单 missingList` | array | `[{table,id}]`，缺失（待补建）项，详情/待补入口直接展示，免再算 |
| `异常清单 abnormalList` | array | `[{table,id,reason}]`，库行存在但内部数据有问题（下游悬空 / 维度不一致），列表「数据异常」橙标来源 |
| `末次校验 verifiedAt` | time | 判断状态是否过期、是否需重算 |

> **对账项三态**：命中（复用现有库行）/ 缺失（待按同规则补建）/ 数据异常（库行存在，但内部数据指向下游不存在项、或维度与本装备不一致）。
> **cfgStatus 优先级**：`待补 > 数据异常 > 已配齐`——有缺失即标 `待补`；无缺失但有异常标 `数据异常`；全部命中且无异常才 `已配齐`。（与 §七、§十二、demo 三态一致）

### 11.3 对账触发时机（只在"写"时算）

| 触发 | 重算范围 | 成本 |
|------|---------|------|
| 新建 / 编辑装备 | 该单件整条链 | O(M)，便宜 |
| 补库 / 改库 / 删库（关联库条目变更） | 经反向索引，**只重算引用该条目的装备** | 正比受影响装备数 |
| 补 item / language | 该单件对应对账项 | 便宜 |
| 全量体检（导入批量 / 改编码规则后） | 全表 | **后台异步**，跑完写回状态，不阻塞页面 |

> 删除一条库条目时，引用它的装备状态可能从 `已配齐` 翻成 `待补`——靠反向索引同步刷新。

### 11.4 增量更新 + 反向索引

- 维护反向索引 `refIndex: { "表名:条目ID" → [引用它的装备ID…] }`。
- 装备**新建**时，把它的每个对账 FK 注册进 `refIndex`。
- 装备**编辑**时：编辑可能改变拼出的 FK（如改品质 → PropLibID 变）。须**先按旧 FK 集合从 refIndex 注销、再按新 FK 集合注册**，否则旧 `旧ID→装备` 映射残留，会在该旧库条目变更时**误触发**本装备重算。注销/注册 + cfgStatus 重算应在**同一事务内原子完成**。
- 库条目变更时，查 `refIndex` 定位受影响装备，**只重算这几件**，而非扫全表。
- 这样"库变更"的代价正比于受影响装备数（通常个位~百级），与装备总量无关。

### 11.5 一件装备的对账项清单（要校验哪些环节）

与 [10 Step2](10_装备配置流程.md) + 配套表一致：

- **关联库**：equip_proplib · equip_job_group · 随机词条库（垃圾/高级/[远古/太古，暂未启用]）· equip_special_drop（带特效，→equip_special_prop→skill）· equip_suit（带套装）· **equip_skill_drop（带装备技能）→ equip_skill_effect（二级，→skill）**
- **配套**：item 非绑 + 绑定 两条
- **文案**：language 名称 + 描述

> ⚠ **地基字典不在对账项内**：Sheet4 / equip_rare / prop / soul / equip_random_proplib_group 是 equip 实时查表的真上游（[10 第三节](10_装备配置流程.md)），假定恒在，不逐件对账。

### 11.6 命中检查：哈希集合 O(1)

- 每个关联库的主键列预载为内存 `Set`；对账 = `FK ID ∈ Set`，单次 O(1)。
- 即便全量体检也是 N×M 次 O(1)，配合 11.3 的异步执行即可承受。

### 11.7 复用分支的"维度一致性核对"（对账要点）

按 [10 文档](10_装备配置流程.md)，命中（复用）不等于万事大吉，还要确认**库行的维度与本装备一致**（别张冠李戴）：

- 单件详情进入时，对每个**命中**的对账项，额外比对"库行记录的维度 vs 本装备维度"，不一致则提示（新悬空根因：维度填错 / 两侧规则不一致 → ID 对上了但语义错了）。
- 维度不一致、以及"库行存在但内部数据下游悬空"的结果，**写入 `abnormalList`**，列表按三态呈现「数据异常」橙标。
- 此核对成本高，**计算仍只在"写时"（新建 / 编辑 / 补库）与单件详情触发**（写时算、读时取），**不在列表加载时实时跑**——列表只读预存的 `cfgStatus / abnormalList`。

### 11.8 呈现与解耦

- **列表**：只读 `cfgStatus`；标记列显示配置状态三态（已配齐 / 待补 N 项 / 数据异常，见 §七）；筛选栏加「配置状态」下拉。
- **待补入口**：用状态筛选 / 红点把"待办工作流"与主列表解耦（类比活动配置「待审核」Tab），待补通常是少数，不污染主列表性能。
- **详情**：进入时跑单件全链对账（顺带 11.7 维度核对），逐环节展示 命中/缺失。

### 11.9 性能量级（为什么不慢）

| 操作 | 复杂度 | 量级感 |
|------|--------|--------|
| 列表加载 | O(1)/行（读字段） | 十万级装备无压力 |
| 单件新建/编辑/详情 | O(M)，M≈6~10 | 毫秒级 |
| 库条目变更 | O(受影响装备数 × M) | 通常个位~百级 |
| 全量体检 | O(N×M)，异步 | 后台跑，不阻塞 |

### 11.10 与 demo 的关系

- demo 已是"列表读 flag（`_incomplete`）"的形态，种子数据默认视为已配齐，**未对每行跑全链**——所以演示层不慢。
- 本节是把同一个 flag 在真实工具里升级为 **持久化字段 + 增量维护 + 反向索引** 的生产级版本。**本节为设计补充，不改 demo。**

## 十三、配表加载与回写（数据生命周期 + 变更记录）★

> 工具**不自带数据库**，而是"**加载本地配表 → 在工具里改 → 写回本地配表**"的会话式批次工具。本节定义这条数据生命周期，以及回写时的**变更记录**，并与现有配表工作流（SVN trunk / ExcelMerge 比对 / `excel-batch-commit`）衔接。

### 13.1 数据生命周期总览

```
① 加载 Load     选本地 xlsx（equip + 关联表）→ 解析 → 按 schema 映射进内存 → 建基线快照 → 初始对账
② 编辑 Edit     增/改装备、补关联库行、改文案…（内存改动，相对基线累积 diff）
③ 比对 Preview  回写前用 ExcelMerge 比对「改动 vs 原表」，用户确认
④ 回写 Write    只把改动写回对应 xlsx 的对应单元格 / 新增行（不碰公式列与格式）
⑤ 记录 Log      生成本次变更记录（改了什么）→ 作为提交说明 →（可选）走 excel-batch-commit 提交 trunk
```

> 与 §十一 的关系：§十一 的配置状态是"一次会话内"的内存态；本节是"会话的两端"——从本地表载入、改完写回本地表。

### 13.2 加载（Load / Import）

- **选源**：选 equip 主表 xlsx，工具按 [09](09_关联表字段清单.md) 的关联关系**带出需要的关联表/公共表**（job_group / proplib / ... / language / prop），或直接选配表目录。
- **解析映射**：按 [13 schema](13_字段配置标准（表与字段Schema）.md) 把每张表的列映射进内存——用 `srcCol`（列坐标）定位、`key` 索引（不受 Remark 撞名影响，见 13 §二）。
- **基线快照（baseline）**：加载后冻结一份原始快照，作为之后"本次改了什么"的 diff 基准。
- **初始对账**：加载即跑一遍 §十一 的对账，给出每件装备初始状态（已配齐 / 待补 / 数据异常）。
- **大表策略**：proplib 等两万+行的大表按需索引 / 懒加载，不全量进内存（见 §11.9）。

### 13.3 回写（Write-back）—— ★ 只改手填，不碰公式

回写是本设计**最关键的约束**：配表里大量列是 **Excel 公式**（equip 30+ 列、关联表也有），工具**绝不能把公式列写成静态值**，否则公式失效、配表彻底失去可维护性。

| schema `source` | 控件 | 回写行为 |
|---|---|---|
| `manual` 手工配置 | select / int-input / resource / switch | **写回**（策划真正改的维度） |
| `formula` 公式生成 / `ref`·`combo`（自洽编码 gen） | gen | **不写**——留空或向上复制同列公式，让 Excel 自己算 |
| `imported`（special_drop 等外部导入） | gen | 不写（外部主表维护） |
| `deprecated` / `hidden` | — | 不碰 |

- **改装备** = 在 equip 行**只更新手填维度单元格**，公式列保持原公式。
- **新增装备** = equip 表**追加一行**，只填手填维度，公式列**向上复制公式**（沿用整列公式）。
- **补关联库行** = 在对应库 xlsx 追加行，填该库 `manual` 字段（库 PK 按编码规则、与 equip 自洽）。
- **改文案** = 写 language 表对应 Key 的中文。
- **保留格式**：按单元格精确改，保留原表公式 / 格式 / 批注 / 未改动内容（**不整表覆盖**）。

### 13.4 变更记录（Changelog）—— ★ 写入即留痕

回写时对 baseline 做 diff，生成**本次变更记录**：

```
本次变更（2026-06-24 14:30 · 策划-小明）
■ equip 装备：新增 2 件、修改 1 件
   + 3011017001  战士0转1级太古武器（新增；维度 武器/战士/0转/分支1/太古/1级）
   + 3011015001  战士0转1级橙武器（新增）
   ~ 3021011001  弓0转1级绿头饰：品质 绿→蓝、几级 1→2
■ 关联库：新增 3 条
   + equip_proplib      101007001（为 3011017001 补建）
   + equip_special_drop 1011011（特效）
   + equip_suit         5005
■ language：新增/改 4 条 Key
   + EquipName_3011017001 = 太古战刃 …
```

- **粒度**：表 → 行（业务 ID + 坐标）→ 字段 → 旧值 → 新值；新增 / 修改 / 删除 分类。
- **用途**：① 写入留痕、可追溯；② **直接作为提交说明**喂给 `excel-batch-commit`；③ 回看本批次做了什么。
- 变更记录**随每次回写生成**，可存为 changelog 文件 + 进工具内"提交历史"。

### 13.5 与现有配表工作流衔接

| 环节 | 复用现有能力 |
|---|---|
| 回写前比对 | **ExcelMerge** 比对「改动 vs 原表」，用户确认后再写（沿用"提交前必须比对确认"约定） |
| 写回 trunk | 变更记录作为规范 log → 对接 **`excel-batch-commit`** skill（自动更新、逐表提交、拼装 log） |
| 多人协作 | 加载时若本地表已被他人更新（SVN），先 update 再 load，避免基于过期基线回写 |

### 13.6 关键约束小结

1. **只写 `manual`，公式列一律不碰**——工具能安全融入现有 Excel 配表流的前提。
2. **回写前必比对**（ExcelMerge），用户确认才落盘。
3. **每次回写必出变更记录**，作提交说明与追溯依据。
4. **基线快照**是 diff 的唯一基准；多人协作先同步再加载。
5. demo **只展示加载/写回入口**（header 两按钮 + 加载弹窗 + 写回弹窗含变更记录预览），**不接真实文件**（内存 mock，见 §十二）；本节为真实工具的数据 I/O 设计。

### 13.7 技术实现参考（对标 ItemEditor 道具编辑器）

> 同目录已有的道具编辑器 `策划work\tools\ItemEditor` 已验证一套"加载本地配表 → 编辑 → 写回"的可行架构，可借鉴；但它的回写方式有三处硬伤，**正是本节（§13.3/13.4）要避开的**。

**可直接借鉴的架构（零后端本地读写闭环）**：
- 纯前端单文件 HTML + 浏览器端 **SheetJS(xlsx.js)** 解析/生成 xlsx；**File System Access API** 直接读写本地文件；**IndexedDB** 持久化"配表目录 / 图标目录"句柄，下次自动恢复。启动只需一个静态 http 服务（`file://` 下 FSA 受限）。
- **4 行表头协议**与本套配表一致（Row1 中文名 / Row2 A·N·S·C / Row3 类型 / Row4 字段名 / Row5+ 数据）——`sheet_to_json({header:1})` 取二维数组，前 4 行建 schema、数据行记 `_rowIndex`（原行号）。这正是 [13 schema](13_字段配置标准（表与字段Schema）.md) 的 `srcCol/srcName` **可从表头自动生成基础结构**的依据。
- 语言表：`langMap = {Id→Zhs}` 全表扫描建映射，UI 显中文、编辑只改 Key（印证本工具"语言包显示中文"，见 §5.3）。
- 图标：File System Access 选图标根目录 → 文件名索引 → 按需 `createObjectURL`，句柄存 IndexedDB 自动恢复（解决 §5.4"待接资源库"）。

**必须避开的三处硬伤（ItemEditor 的回写缺陷）**：

| ItemEditor 怎么做 | 问题 | 本工具的改进 |
|---|---|---|
| `aoa_to_sheet` **整表重建**写回 | 公式/格式/批注全丢，多 sheet 时其余 sheet 也丢 | **逐单元格回写**：加载原 workbook，只 set 改动的 cell；公式列（source≠manual）一律不碰（§13.3） |
| 直接覆盖原文件 | **无备份** | 写回前自动存带时间戳备份 |
| 仅会话级 `undoStack`，保存即清空、不落盘 | **无持久变更记录** | 把 `{行ID·字段·旧值→新值}` 落成 changelog（§13.4）——它的撤销栈结构 `{item,field,oldValue,label}` 正是现成数据源，只差持久化 |

**其它经验（其开发纪要）**：列表必须分页（它用每页 200 条）；循环内禁用 `indexOf`（用 Map 预建行号索引，避免 O(n²)）；统一 Excel 整数 vs 字符串 "1" 的类型偏差；多 sheet 必须**全部回写**（它只写第一个 sheet 导致丢数据，是其明确未修的坑）。

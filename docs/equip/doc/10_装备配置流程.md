# 10 · 装备配置流程（配一件装备要做什么）

> 面向**要新配/改一件装备**的策划。把"装备配置"拆成可执行步骤：先填什么、哪些自动、关联库怎么对账（复用还是补建）、还要连带配哪些表。
> 字段细节见 [05 字段清单](05_字段清单表.md)，关联表见 [09](09_关联表字段清单.md)，编码规则见 [04](04_ID编码规则速查.md)，整体关系见 [08 ER图](08_ER关系图.md)。

---

## 一、先建立三个核心认知

1. **equip 主表是"半自动"的**：你只手填几个"维度"（部位/职业/转数/品质/等级…），装备ID、属性库ID、职业限制、名称/描述等几十列都由公式自动算出。

2. **关联不是"先建后引用"，而是"同规则共生、ID 对齐"**：equip 的关联字段(FK) 和 关联库的对应主键(PK)，是**同一套拼接规则各跑一遍**的产物（如 `属性库ID = 物理魔法+部位+转数+品质+转数等级`，equip 拼一次、equip_proplib 也按同规则拼一次）。**equip 填维度时根本不读关联库**——所以 equip 能先行，库里有没有那条，不影响 equip 出 ID。

3. **顺序不是重点，"命中与否"才是**：equip 拼出的那个 ID，**既是外键、也是查询键**。拿它去库里查——
   - **命中** → 复用现有库行（什么都不用建）；
   - **缺失** → 按同规则在库里补一条。
   这一步叫**对账**，复用和新建只是它的两个分支。

> 所以本流程的主轴不是"自下而上建库"，而是：**equip 先出 ID → 逐 FK 对账（命中复用 / 缺失补建）**。

---

## 二、配置顺序总览（对账模型）

```
① 地基字典   实时被 equip 公式查，常规装备一般不动
            equip内置Sheet4 · equip_rare · prop · soul · equip_random_proplib_group
② equip 主表  ★先行：填维度 → 自动拼出全部 FK ID
③ 逐 FK 对账  ★核心：每个自动ID 去对应库查 —— 命中→复用 ／ 缺失→按同规则补建
            关联库：proplib · job_group · random_proplib · special_drop+prop · suit · skill_drop+effect
④ item 配套   给装备配"道具"（非绑+绑定两条，基本逐件新建）
⑤ language    补名称/描述/套装名 多语言文案
```

> 和"自下而上"的差别：**②③ 不再是"先建库再填 equip"，而是 equip 先行、库在对账时按需复用或补建**。地基字典(①)因为是被实时查表的真上游，仍然始终先就位。

---

## 三、地基字典：唯一"必须先于 equip"的一类

只有这几张是 equip（或关联库）的**公式实时去读**的真上游，必须先就位。配常规装备一般已存在、不用动：

| 表 | 谁在实时查它 | 何时才需要动 |
|----|-------------|-------------|
| equip 内置 Sheet4 字典 | equip 自身（部位名/品质名/真实等级/套数/命名词/文案…） | 新部位、新等级段、改命名规则 |
| equip_rare（品质表） | equip 的 道具品质/强度子品质 实时查它 | 调整品质体系 |
| prop（属性主表） | equip_proplib / equip_random_proplib 反查"属性名→编号" | 新增一种全新属性 |
| soul（职业总表） | equip_job_group 汇总职业ID列表 | 新增职业或转职分支 |
| equip_random_proplib_group（品质乘算） | equip_random_proplib 按品质取浮动比例 | 调随机词条浮动规则 |

> 这几张是"地基"，影响面大、改动谨慎；它们与 equip 是**真·查表依赖**，不在"对账"范围内。

## 四、分步操作

### Step 1 · 在 equip 主表新增装备行，填"维度"

只需手填这几列（其余公式自动出）：

| 必填  | 字段                                 | 说明                   |
| --- | ---------------------------------- | -------------------- |
| ✅   | 装备部位(EquipType)                    | 1武器…11项链             |
| ✅   | 基础职业(Remark3)                      | 1战 2弓 3法 4牧          |
| ✅   | 转数(TransferNum)                    | 0~4                  |
| ✅   | 1-2转分支职业(Remark6)/3转分支职业(Remark28) | 分支编号                 |
| ✅   | 品质(Remark8)                        | 1绿~7太古               |
| ✅   | 几级的装备(Remark10)                    | 同转数下第几套              |
| ✅   | 挂件ID(AccessoriesID)                | 武器/饰品外观（衣服走模型组，自动）   |
| ✅   | 装备图标(Remark27)                     | icon 资源名             |
| ⭕   | 初始套装库(SuitId)                      | 属于套装才填，指向 equip_suit |
| ⭕   | 装备技能随机库(RandonSkillID)             | 需要随机技能才填             |
| ⭕   | 装备描述内容(Remark26)                   | 想覆盖默认文案时手写           |

填完后，装备ID / 种类 / 使用职业限制 / 真实等级 / 穿戴等级 / 初始基础属性库id / 初始随机属性库id / 特殊属性库ID / 名称描述语言Key 等都会**自动生成**——这些就是下一步要对账的 FK。

### Step 2 · 逐 FK 对账（★核心：命中复用 / 缺失补建）

把 equip 自动拼出的每个 FK ID，拿去对应库查一次：

| equip 自动 FK | 去这张库对账 | 默认倾向 | 缺失时补建的关键字段 |
|--------------|------------|---------|--------------------|
| 使用职业限制(UseJobRequest) | equip_job_group · GroupID | **高复用**（就那几十组） | 组ID、职业ID(可穿列表)、基础职业/转数/分支 |
| 初始基础属性库id(PropLibID) | equip_proplib · PropLibID | 偏复用 | 属性库ID(同规则)、属性值"属性ID,数值" |
| 初始随机属性库id 里各库ID | equip_random_proplib · RandomPropLib | **高复用**（按部位） | 随机库ID、属性种类、属性id、基准值、权重 |
| 特殊属性库ID(DropID)（带特效时） | equip_special_drop · DropID（→ equip_special_prop） | 看情况 | 库ID、属性ID(→special_prop)、权重；特效本体在 special_prop |
| 初始套装库(SuitId)（套装时） | equip_suit · SuitId | 看情况 | 套装编号、套装名、套装总数 |
| 装备技能随机库(RandonSkillID)（带技能时） | equip_skill_drop · SkillGroup（→ equip_skill_effect） | 看情况 | 技能组ID、候选技能权重、关联效果编号；效果本体在 skill_effect |

**两个分支怎么做**：

- **命中（复用）**：库里已有同 ID 的行——**不用建**，只确认它的维度和这件装备一致（别张冠李戴）。
- **缺失（补建）**：库里没有——在库里**填同一组维度**，让它按同规则自拼出**相同的 PK**（不是手抄 equip 的 ID）。补完再查一次应命中。

> ⚠️ 悬空外键的根因，从旧版的"漏建库"变成了：**某侧维度填错 / 两侧规则不一致 → ID 对不上**。排查方向也随之从"库里有没有"变成"两侧是不是同一组维度"。

#### 体检表：把对账做成自动标红（强烈建议）

命中与否就是**纯字符串相等**判断，可半自动化。在 equip 表（或单开一张校验页）对每个 FK 加一列命中标记，**填到单元格时在公式前加 `=`**（本文档为防飞书误判公式，示例去掉了开头的等号）：

```
命中标记列： IF( COUNTIF( 目标库主键列 , 本行FK单元格 ) > 0 , "✓" , "✗缺" )

例（校验 PropLibID）：
   IF( COUNTIF( [equip_proplib.xlsx]Sheet1!$A:$A , $X5 ) > 0 , "✓" , "✗缺" )
```

- 复用的自动 ✓、漏建的自动 ✗缺，一眼看到要补哪条；
- 维度填错把 ID 拼歪的，也会 ✗，顺带抓出来；
- 每个 FK 一列，全绿才算无悬空外键。

### Step 3 · 配套：在 item 表加两条道具

装备在背包/交易里是"道具"，每件装备要在 item 配 **非绑 + 绑定** 两条（详见 [09 表12](09_关联表字段清单.md)）。这步**基本是逐件新建**：

| | 道具ID(ItemId) | 绑定物品ID | 未绑定物品ID | 所在页签 |
|---|---|---|---|---|
| 非绑 | = 装备ID | = "8"+装备ID | 空 | 「XX装备」 |
| 绑定 | = "8"+装备ID | 空 | = 装备ID | 「XX装备-绑定装备」 |

- 道具ID 与装备ID 同源（同规则），名称语言Key 与 equip 共用；两条道具名称/品质/图标/出售返还大多相同。
- 例外的"复用"：道具出售返还(SellItems) 里引用的货币道具(如 1000金币) 是**复用** item 表里已有的货币行，不用新建。
- 装备页里整列为空的字段已弃用，不用填（见 [09 表12 的弃用标注](09_关联表字段清单.md)）。

### Step 4 · 补文案：language 表

把 equip 自动生成的名称Key(`EquipName_装备ID`)、描述Key(`EquipDes_装备ID`)，以及套装名(`Equip_SuitName_套装ID`)、装备技能套装名(`SuitName_xxx`)等，在 language 表填上对应中文。

### Step 5（按需）· 特效 / 套装 / 装备技能 的"开关"

- **要特效**：equip「备注(Remark)」加"特效"字样 → 自动生成 特殊属性库ID(DropID) → 回 Step2 对 equip_special_drop / equip_special_prop 对账补建。
- **要套装**：备注加"套装"字样 → 自动生成 图鉴套装ID；并填 初始套装库(SuitId)，对 equip_suit 对账补建。
- **要装备技能**：填 装备技能随机库(RandonSkillID) → 对 equip_skill_drop / equip_skill_effect 对账补建。

---

## 五、配置检查清单

- [ ] equip 维度列填全（部位/职业/转数/分支/品质/第几套/挂件/图标）。
- [ ] **逐 FK 对账通过**：每个自动 ID 在对应库都命中（体检列全 ✓），缺失的已按同规则补建。
- [ ] 复用的库行维度与本装备一致（没张冠李戴）。
- [ ] item 配了非绑+绑定两条，ID 差"8"前缀、L/M 对填。
- [ ] language 补齐名称/描述（及套装名）文案。
- [ ] 带特效/套装/装备技能的：备注关键字写了、对应库对账补建了。
- [ ] 随机词条条数符合品质预期（见 [05 随机词条条数对照表](05_字段清单表.md)）。

---

## 六、一句话流程

**地基字典就位 → 在 equip 填维度（其余自动出 ID）→ 逐 FK 对账：命中复用、缺失按同规则补建（体检列标红兜底）→ 在 item 配套两条道具 → 补 language 文案 →（按需）开特效/套装/技能开关。**
